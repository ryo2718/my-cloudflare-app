# 設計判断の記録 (Design Decisions)

このドキュメントは、プロジェクトの主要な設計判断と**その理由**を記録する。
将来「なぜこうなってるんだっけ？」となった時の参照用。

---

## 全体方針

### 計算ロジックと表示ロジックの分離

**判断**: JSONを契約として、計算側と表示側を完全に独立させる。

**理由**:
- 表示側はダミーデータで先行開発できる
- 計算側を後から差し替え可能 (Desktop Postflop → 別ソルバー → 自前ヒューリスティック)
- データを手動編集してテストできる
- スキーマ駆動開発の王道パターン

**影響**: このリポジトリには計算ロジックを書かない。表示側に集中する。

---

## ソルバーの選定

### Desktop Postflop を採用予定

**判断**: Desktop Postflopを将来の計算側で使用予定 (このリポジトリには含めない)。

**比較した選択肢**:

| ソルバー | 精度 | 速度 | ライセンス | 結論 |
|---|---|---|---|---|
| **Desktop Postflop** | ★★★★★ | ★★★★★ | AGPL-3.0 | **採用** |
| WASM Postflop | ★★★★★ | ★★★★ | AGPL-3.0 | 自動化に不向き |
| TexasSolver | ★★ | ★★ | AGPL+独自条項 | 精度不安・追加制限 |
| PioSOLVER Free | ★★★★ | ★★★ | プロプライエタリ | 機能制限多 |
| GTO Wizard | ★★★★★ | - | スクレイピング不可 | **規約違反、論外** |

**Desktop Postflopを選んだ理由**:
- 精度: PioSOLVER/GTO+とほぼ同じ結果
- 速度: WASM版の約2倍速、PioSOLVERより速いケースあり
- ライセンス: 標準AGPL-3.0 (TexasSolverのような追加制限なし)
- CLI自動化が可能 (1755フロップのバッチ処理に必要)

---

## ライセンス・法的観点

### AGPL-3.0の扱い

**判断**: ソルバー本体をアプリに含めない。計算結果のJSONだけ配信する。

**理由**:
- AGPLは「ソフトウェア自体」に適用される。計算結果の数値データは事実情報
- ソルバーをWebアプリに組み込むとAGPL感染 → アプリ全体のソース公開義務
- 計算結果データだけならライセンス自由、アプリのライセンスも自由

**やってはいけないこと**:
- ❌ Desktop PostflopのコードをWebアプリにバンドル
- ❌ ブラウザ上でソルバーを動かす (WASM Postflopをアプリに組み込む等)
- ❌ GTO Wizardからのデータスクレイピング (規約違反)

**やっていいこと**:
- ✅ 自分のPCでDesktop Postflopを動かして計算
- ✅ 計算結果のJSONをアプリにバンドルして配信
- ✅ アプリのライセンスは自由 (MIT, 商用、何でも)

## ホスティング

### Cloudflare Pages

**判断**: Cloudflare Pagesに無料デプロイ。

**理由**:
- 帯域**無制限** (50人規模なら何があっても無料)
- CDN経由で世界中から高速アクセス
- GitHubと連携で自動デプロイ
- カスタムドメイン無料対応

**比較**:

| サービス | 帯域 | 50人運用 |
|---|---|---|
| **Cloudflare Pages** | 無制限 | ◎ |
| GitHub Pages | 100GB/月 | ○ |
| Vercel | 100GB/月、商用注意 | △ |
| Netlify | 100GB/月 | ○ |

## データ設計

### ハンド粒度: 169通り (プリフロップ)

**判断**: プリフロップは169通り、フロップ以降は1225組み合わせ。

**理由**:
- プリフロップ: ボードがないので、組み合わせ (AsKh vs AsKd) を区別する戦略的意味がない
- フロップ以降: ボード3枚との関係 (バックドアフラッシュなど) で組み合わせを区別する必要あり

### EVを含めない

**判断**: 戦略確率のみ持つ。EV (期待値) は含めない。

**理由**:
- データサイズ削減 (約2倍違う)
- 表示の優先度: まずは戦略を見たい、EVは後で
- 必要になったらスキーマv2で追加可能

### ファイル粒度: 1シナリオ=1ファイル

**判断**: `rfi_utg.json`, `3bet_btn_vs_utg.json` のように細かく分割。

**理由**:
- 遅延ロード可能 (見るシナリオだけfetch)
- CDNキャッシュ効率良い
- ファイル名でシナリオが識別可能
- 全部を1ファイルにすると初期ロード重い、編集も困難

### ファイル命名規則

**判断**: `アクション_ポジション[_vs_相手].json` (アクション先頭)

**例**:
- `rfi_utg.json`
- `3bet_btn_vs_utg.json`
- `vs_3bet_utg_vs_btn.json`

**理由**:
- アクションでソートしやすい (RFI, 3bet, 4bet... の順に並ぶ)
- ポジション順でファイル名が並んで見やすい

**却下した案**:
- `UTG_RFI.json` (大文字派) - ソートしにくい
- `00_UTG_RFI.json` (プレフィックス) - 拡張時に番号振り直し必要

### 3betの両視点を作る

**判断**: 3betする側 (`3bet_btn_vs_utg.json`) と されたほう側 (`vs_3bet_utg_vs_btn.json`) の両方を作る。

**理由**:
- 自分の視点だけでなく、相手のレンジを推測したい時にも使う
- ファイル数は倍になるが、データの重複ではなく異なる戦略

## 技術スタック

### React採用

**判断**: フレームワークはReact。

**理由**:
- AIアシスト (Claude/Claude Code) の精度が一番高い
- エコシステムが圧倒的 (大量リスト表示の `react-window` 等)
- 拡張性 (将来クイズ機能、ユーザー機能の追加が楽)
- 求人・情報量で他より優位

**比較した選択肢**:

| | React | Vue | Svelte | プレーンJS |
|---|---|---|---|---|
| 学習コスト | 中 | 低 | 低 | ほぼゼロ |
| AIアシスト | ◎ | ◎ | ○ | ◎ |
| 拡張性 | ◎ | ◎ | ○ | △ |
| コード量 | 多 | 中 | 少 | 多 |

Svelteは魅力的だがAIアシストの精度でReactに劣るため却下。
プレーンJSは13×13マトリクス + 状態管理で確実に複雑化するため却下。

### Vite採用

**判断**: ビルドツールはVite。

**理由**:
- 高速開発サーバー
- ビルド設定が最小
- React公式の推奨スタック (CRAは非推奨化)
- Cloudflare Pagesとの相性◎

### スタイリング: 当面インラインstyle

**判断**: 雛形ではインラインstyle、将来Tailwind検討。

**理由**:
- 雛形フェーズではシンプルさ優先、外部依存最小
- artifactとしてプレビュー確認したかった
- プロダクション化時にTailwindに移行可能

## 将来の拡張ポイント

### フロップ対応 (フェーズ2)

**未確定**:
- ベットサイズ何種類か (3 vs 4 vs 5)
- ツリーの深さ (3 vs 4 vs 5)
- 全1755フロップ vs 代表184クラス

**マシンスペック制約**: i7-12700KF / 64GB RAM
- 4サイズ × 深さ3: 全1755フロップを2-3日で計算可能
- 4サイズ × 深さ4: 184クラスに絞れば12時間で完了
- 5サイズ以上: 非現実的

### マルチスタック対応

**判断**: まず100bbのみ。複数スタックは将来。

**実装時の方針**:
- ファイル名にスタック情報含める: `rfi_utg_100bb.json`
- またはディレクトリで分ける: `data/100bb/rfi/...`, `data/50bb/rfi/...`
- 後者を推奨 (ファイル名がシンプルに保てる)

### ヘッズアップ対応

**判断**: まず6max。HUは将来。

**実装時の方針**: `metadata.table_size` で識別、ポジションは `BTN`/`BB` の2種のみ。

### UX: アクション履歴のベットサイジング表示 (優先度: 低)

**未着手**。各ペインの subtitle 等に、現在ノードまでのベットサイジング履歴を表示する。

**例**: `BB 3-bet to 12 / UTG 4-bet to 22 / BB 5-bet to 34 (1.55x)`

**動機**: 5-bet 等の深いノードで「AA が 79% Call」のような GTO 出力が、サイジングツリー
(極端に小さい 5-bet → SPR圧縮 → コール優位) の結果であることを直感的に理解しづらい。
`game_info.action_history` を整形表示するだけで実装可能。

**実装スケッチ**:
- `src/utils/format.ts` 等に `formatActionHistory(history): string` を新設
- `DualRangeView` の `subtitle` か `HandDetail` 上部のメタ行で表示
- 直前 raise からの倍率 (1.55x 等) を併記すると「圧縮されている」が伝わりやすい

## UI 表示モード

### 「2レンジ並列表示」の採用 (2026-04-27)

**判断**: ScenarioSelector を「Opener / Responder の 2 ドロップダウン」+ 「左右並列マトリクス」に書き換えた。RFI / vs RFI のカテゴリタブは UI から削除。

**理由**:
- 学習用途では「opener のレンジ」と「responder の応答レンジ」を同時に見比べたいケースが多い
- 単独カテゴリ切替よりも対比表示の方が示唆的 (例: UTG オープン頻度と BB の 3bet 頻度を並べると、対応関係が掴みやすい)
- カテゴリタブが 1 つだけアクティブで他が disabled になるよりはシンプル

**保持したもの (将来の拡張余地)**:
- `src/data/scenarios.ts` 内の `SCENARIO_CATEGORIES`, `CategoryId`, `CategoryDef` データ構造は残してある
- 3bet / 4bet / 5bet レベルのデータが追加された場合は、別途「アクションレベル」セレクタの追加か、本UIの拡張 (例: 3bet レンジ → vs 3bet レンジを並列表示) を**改めて再設計**する必要あり。今は YAGNI でその時点まで設計を保留

**新規ヘルパー** (`src/data/scenarios.ts`):
- `POSITION_ORDER`: アクション順 (UTG → BB)
- `OpenerPosition`, `OPENER_POSITIONS`: BB を除いた 5 ポジション
- `getValidResponders(opener)`: opener より後ろのポジション (responder ドロップダウン候補)
- `getRfiPath(opener)`, `getVsRfiPath(opener, responder)`: JSON ファイルパス
- `getRfiScenario(opener)`, `getVsRfiScenario(opener, responder)`: Scenario オブジェクト生成
